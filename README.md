# Bingo Live — backend (virtual currency)

A real database + API backend for the Bingo mini-app, built for a Telegram bot.
**All currency here is virtual/play points — there is no real-money deposit,
withdrawal, or payment processing anywhere in this project.**

## Stack

- **PostgreSQL** — works with any managed free tier (Supabase, Neon, Railway, etc.)
- **Node.js + Express** — REST API, polled by the frontend every ~1s during a live round
- No WebSockets, no payment gateway, no external services required

## 1. Set up the database

Create a free Postgres instance (Supabase and Neon both have a no-cost tier),
then run the schema once:

```bash
psql "$DATABASE_URL" -f schema.sql
```

This creates 6 tables:

| Table | Purpose |
|---|---|
| `players` | One row per Telegram user, holds `main_wallet` (winnings) and `play_wallet` (starting balance, default 1000 points) |
| `rounds` | One row per game round: stake, status, derash, timing |
| `round_cards` | Which card number(s) each player picked for a round, with the generated 5×5 layout stored |
| `round_calls` | The ordered sequence of numbers called in a round |
| `transactions` | Full ledger — every wallet change (stake debits, win credits) is a row here |
| `round_winners` | Which card won, how (row/col/corners/diagonal), and the payout |

## 2. Configure and run the server

```bash
cp .env.example .env
# fill in DATABASE_URL and BOT_TOKEN
npm install
npm start
```

`BOT_TOKEN` is your bot's token from @BotFather — it's used to verify that
requests genuinely came from your Telegram Mini App (see `src/telegramAuth.js`),
not to send messages, so no Telegram Bot API calls happen here.

## 3. How it fits together with the Telegram bot

1. In BotFather, set your Mini App URL to wherever you host the **frontend**
   (the `bingo.html` file) — e.g. `https://your-frontend.vercel.app`.
2. That page loads `https://telegram.org/js/telegram-web-app.js` and, on load,
   sends `Telegram.WebApp.initData` to `POST /api/auth/telegram` on **this**
   backend (hosted separately, e.g. Railway/Render/Fly.io).
3. The backend verifies the signature, upserts the player, and returns their
   `player.id` + wallet balances — the frontend uses that `id` for every
   subsequent call.

## 4. API summary

| Method & path | Purpose |
|---|---|
| `POST /api/auth/telegram` | Verify Telegram initData, upsert player, return player + wallets |
| `GET /api/wallet/:playerId` | Current Main/Play wallet balances |
| `GET /api/players/:playerId/transactions` | Wallet ledger history |
| `GET /api/players/:playerId/rounds` | Past rounds this player took part in |
| `GET /api/rounds/active?stake=10&player_id=` | Get (or create) the current waiting round for a stake, with taken cards |
| `GET /api/rounds/:roundId/state?player_id=` | Poll this during a round: called numbers, current call, your cards + win status |
| `POST /api/rounds/:roundId/select-card` | Body `{player_id, card_number}` — validates funds & 2-card max |
| `POST /api/rounds/:roundId/deselect-card` | Body `{player_id, card_number}` |

## 5. Game rules encoded in the backend (authoritative — not trusted from the client)

- Max **2 cards** per player per round, from **600** possible card numbers
- Round needs **≥10 distinct players** with a card to start; otherwise the
  countdown extends by 15s and rechecks (matches the frontend's "Waiting for
  more players…" behavior)
- **Derash** (prize pool) = `cards_sold × stake × 80%` (20% house cut)
- Stake is debited from **Play Wallet** first; any shortfall is drawn from
  **Main Wallet** automatically
- A card wins on: a full **row**, a full **column**, **both diagonals**, or
  **all four corners** — the center cell counts as automatically marked (FREE)
- If multiple cards complete on the same call, the Derash is split evenly
  and each share is credited to that player's **Main Wallet**
- Card layouts are generated by a seeded RNG (`src/rng.js`) keyed off the
  card number, so card #237 always looks identical — this file is a direct
  port of the same algorithm in `bingo.html`, so previews and the
  server-authoritative card always match

## 6. `bingo.html` is now fully wired to this backend

Nothing in the frontend is simulated anymore — no `Math.random()` calls,
no local `mainWallet`/`playWallet` variables, no fake "taken" cards.
Everything comes from the API:

- On load, it authenticates via `POST /api/auth/telegram` (using
  `Telegram.WebApp.initData`) and shows your real wallet balances
- Choosing a stake calls `GET /api/rounds/active` to join the real current
  round for that stake, and polls `GET /api/rounds/:id/state` every 2s
  while picking (live taken-card list + live "X/10 players joined" count)
- Selecting/deselecting a card calls `POST .../select-card` and
  `.../deselect-card` — any rejection (insufficient balance, card taken,
  max reached) is shown inline, straight from the server's error message
- Once the round goes `active`, it polls `GET /api/rounds/:id/state` every
  1.2s for the live board, current call, your card(s) with called numbers
  marked, and win detection
- A win pulls the real payout + exact winning line (`win_detail`) from the
  server, shows the celebration overlay, refreshes your wallet, and moves
  you back into a fresh picker round automatically
- The Wallet and History tabs load real data from
  `GET /api/players/:id/transactions` and `GET /api/players/:id/rounds`

**Before deploying**, open `bingo.html` and set the one config line near the
top of the `<script>` block:

```js
const API_BASE = 'https://YOUR-BACKEND-URL'; // <-- set this to your deployed backend
```

To test outside Telegram during development, open the page with
`?dev_id=12345` in the URL and set `SKIP_TELEGRAM_VERIFY=true` in the
backend's `.env` (never enable that in production).

## 7. Known simplifications (fine for a first version, worth hardening later)

- Round timers live in server memory (`roundsManager.js`'s `timers` map) —
  a server restart mid-round loses that timer. Add a small reconciliation
  job on boot that resumes any `waiting`/`active` round it finds.
- No rate limiting / abuse protection on the API yet.
- Polling instead of WebSockets is simpler to deploy but slightly less
  snappy than push updates — fine for a 1.2–1.4s call cadence.
- Row/column win highlighting relies on `round_winners.win_detail`
  (added specifically so the UI can show the exact winning line, not just
  "you won a row").
